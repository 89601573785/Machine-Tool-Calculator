## Приоритет 1 — комплексы станков (карточки «линия / комплекс»)

### Задача

В каталоге есть карточки, где указано, что это **комплекс станков** (или готовая лесопильная линия). При перетаскивании на рабочую область должна **сразу формироваться схема** из входящих в комплект станков — не одна карточка, а набор узлов с **готовыми связями** (и конвейерами между ними, см. приоритет 2).

### Сейчас

Перетаскивание любой позиции создаёт **один** `placed-equipment` с одним `equipmentId`.

### Нужно

1. В данных каталога — тип позиции и состав комплекса:

```json
{
  "catalogType": "machine",
  "catalogType": "conveyor",
  "catalogType": "equipment_complex",
  "complex": {
    "name": "Линия …",
    "members": [
      { "catalogId": 12, "offsetX": 0, "offsetY": 0 },
      { "catalogId": 5, "offsetX": 320, "offsetY": 0 }
    ],
    "connections": [
      {
        "fromMemberIndex": 0,
        "toMemberIndex": 1,
        "conveyorCatalogId": 45,
        "defaultConveyorCatalogId": 45
      }
    ]
  }
}
```

2. **Drop комплекса** → функция `expandComplex(complex, dropX, dropY)`:
   - разместить все станки с относительными координатами;
   - выдать каждому свой `placementId`;
   - создать связи между ними с конвейерами из шаблона;
   - сгруппировать визуально (опционально: общая рамка «Комплекс …»).

3. В каталоге на карточке комплекса — бейдж **«Комплекс станков»** и краткий состав («3 станка, 2 конвейера»).

4. После разворота схему можно **редактировать**: двигать станки, менять конвейеры, добавлять/удалять связи.

5. Сохранение в JSON: обычные `placed` + `connections` (комплекс не хранится как монолит, только как набор узлов).

### Чеклист

- [ ] Поле `catalogType` + `complex` в БД / экспорт в JS
- [ ] Маркировка карточек комплексов в каталоге
- [ ] `expandComplex()` при drop
- [ ] Загрузка проекта восстанавливает развёрнутую схему
- [ ] Расчёт и отчёт учитывают все станки и конвейеры комплекса

## Приоритет 2 — связи через конвейеры из каталога

### Задача

Связь между станками — это не «пустая стрелка», а **участок с конвейером** из каталога. Пользователь **выбирает**, какой конвейер использовать (ленточный транспортёр, рольганг, nakopitel и т.д. — позиции уже есть в каталоге).

### Сейчас

- Shift + клик по двум станкам → зелёная ортогональная линия со стрелкой;
- конвейер не указан;
- в модели связи нет `conveyorId`.

### Целевой сценарий

**Вариант A (рекомендуемый):** оставить **стрелки/линии**, но изменить их смысл и оформление:

1. Пользователь инициирует связь (Shift + клик или «Связать» после выбора двух станков).
2. Открывается **модал / панель «Выбор конвейера»** — список из каталога (`catalogType: conveyor`), поиск, превью (название, длина, скорость, цена).
3. После выбора рисуется связь:
   - линия между станками (можно оставить ортогональную трассу);
   - **подпись на связи:** название конвейера, например `Ленточный транспортёр`;
   - **иконка/бейдж «конвейер»** на середине ребра (не только arrowhead);
   - цвет/стиль отличный от «просто связи» (например, оранжевый или серый блок + стрелка направления потока).

**Вариант B:** вместо одной линии — **отдельный элемент «конвейер»** на canvas между станками (прямоугольник с названием). Стрелки только на вход/выход конвейера. Тяжелее в реализации; A проще и ближе к текущему коду.

### Модель данных (JSON v2)

```json
{
  "connections": [
    {
      "fromId": 1,
      "toId": 2,
      "conveyorCatalogId": 45,
      "conveyorName": "Ленточный транспортёр",
      "throughput": null
    }
  ]
}
```

### UI выбора конвейера

- вкладка **«Конвейеры»** в левом каталоге (отдельно от станков и комплексов);
- при создании/редактировании связи — тот же список в модале;
- **«Конвейер по умолчанию»** в настройках (запоминать последний выбор);
- для комплексов — конвейеры проставляются из шаблона, но **можно заменить** (клик по связи → «Сменить конвейер»).

### Отображение на canvas (доработка стрелок)

| Было | Станет |
|------|--------|
| Зелёная линия + arrowhead | Линия + **метка конвейера** по центру ребра |
| Без текста | Подпись: `{название конвейера}` |
| ПКМ — удалить | ПКМ — удалить / **изменить конвейер** |
| Один стиль | Стили: `connection--with-conveyor`, пунктир если конвейер не выбран (черновик) |

### Расчёт и отчёт

- строка в отчёте на **каждый конвейер** в связях: название, catalogId, стоимость, длина/скорость (если есть);
- пропускная способность ребра: `min(станок_от, станок_к, конвейер)`;
- суммарная стоимость линии = станки + конвейеры.

### Правила `canConnect`

- совместимость **`output_type` → `input_type`** (см. приоритет 0);
- запрет дублирующих операций;
- правило «1 ряд» — пересмотреть или оставить дополнительно;
- **без выбранного конвейера** связь можно сохранить как черновик (предупреждение в UI), но в отчёте — «конвейер не указан».

### Чеклист

- [ ] Раздел каталога «Конвейеры», `catalogType: conveyor`
- [ ] Модал выбора конвейера при создании связи
- [ ] `conveyorCatalogId` в `connections` + сохранение/загрузка проекта
- [ ] Подпись и стиль связи «это конвейер»
- [ ] Редактирование конвейера на существующей связи
- [ ] Учёт конвейеров в расчёте и PDF

## Приоритет 0 — аудит и переформирование данных каталога

**Задача:** сначала понять полноту данных по категориям, зафиксировать обязательные поля, заполнять только обоснованные значения.

**Инструмент:** `python scripts/audit-data-completeness.py`

### Снимок (70 записей)

| Поле | Заполнено |
|------|-----------|
| productivity, габариты, power | 100% |
| cost | ~56% |
| efficiency, input/output materials | 0% в БД |
| cycle_time, speed | 0% |

### Алгоритм

1. Аудит → матрица «категория × поле × %»
2. Обязательные поля по категории (`docs/data-schema-by-category.md`)
3. Первичные / производные / не заполнять
4. Экспорт `factory.db` → `equipment-data.js`

### Тип входа и тип выхода

У каждого станка:

```json
{
  "input_type": "brevno",
  "output_type": "lafet"
}
```

Справочник: `docs/material-types.md`. В `canConnect` — совместимость типов. На карточке: «Вход: бреvno → Выход: lafet».

### Категории каталога

Привести `category` к структуре сайта ЛЕСКОМ; разобрать «Прочее» (28 позиций); выделить **конвейеры** и **комплексы** отдельными типами.

## Порядок работ

```
1. Аудит данных + catalogType (machine / conveyor / equipment_complex)
2. Разворот комплексов при drop (expandComplex)
3. Связи с выбором конвейера + подпись на ребре + JSON v2
4. canConnect по input_type / output_type
5. Расчёт и отчёт (станки + конвейеры, min по цепочке)
6. Редизайн UI (sidebar: Станки | Конвейеры | Комплексы)
7. vendor/, доработки интеграции ЛК
8. Тесты
```
